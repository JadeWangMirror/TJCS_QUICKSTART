# TJCS QUICKSTART 整理协作规范

本仓库用于维护同济大学计算机科学与技术专业课程资料。整理目标是让每门课程目录具备统一入口、清晰分类和可持续补充的结构。

## 基本原则

- 保持顶层结构为 `学期/课程名/`，不要把课程资料跨学期或跨课程混放。
- 每门课程目录必须有一个 `README.md`，作为该课程的唯一说明入口。
- 课程目录内零散的说明性 Markdown 应合并进课程 `README.md`；合并后如无独立保留价值，应删除原说明文件。
- 移动文件时尽量保留原文件名中的语义信息，不为了统一而抹掉年份、教师、章节、实验编号、组别等关键信息。
- 不确定资料类型或归属时，先保留在原位置，并在课程 `README.md` 的对应章节写 `TODO` 或在“备注”中说明，不要武断移动。
- 大文件、压缩包、可执行文件和工程产物移动前应谨慎判断，避免破坏项目可运行性。
- 课程设计、大作业、完整项目等项目源代码不直接提交到本仓库；应在课程 `README.md` 中以外部仓库链接、发布页链接或说明链接的形式维护。

## 推荐课程目录结构

每门课程整理后优先使用以下结构。没有对应资料时可以不创建空目录。

```text
学期/课程名/
├── README.md
├── doc/
│   ├── textbook/      # 教材、参考书、讲义
│   ├── ppt/           # 课件、课堂 PPT
│   ├── notes/         # 笔记、复习资料、知识点整理
│   ├── exam/          # 期末、期中、考纲、试卷、答案
│   └── lab/           # 实验指导、实验报告、实验材料说明
├── project/           # 项目说明、外部项目链接索引，不存放项目源代码
├── code/              # 独立代码、脚本、实验代码
├── data/              # 数据集、表格、样例输入输出
├── image/             # 图片、截图、图表
├── video/             # 演示视频、录屏
└── archive/           # zip/rar/7z 等归档包或暂不拆分的历史材料
```

如果课程资料涉及可运行项目，项目源代码应放在独立仓库，并在课程 `README.md` 或 `project/README.md` 中提供链接。本仓库只保留课程说明、项目报告、项目展示文档、演示视频和必要的非源码资料。

## `README.md` 内容规范

课程 `README.md` 建议按以下顺序组织。没有信息的章节可以省略，但不要写空标题。

```markdown
# 课程名

## 课程概述

## 课程工具

## 资料结构

## 课程项目参考

## 实验与作业

## 期末考试与复习

## 备注
```

整理时应优先把原有说明文件中的有效内容完整迁移到这些章节中，避免只留下文件列表。外部仓库链接、课程体验、工具安装说明、考试范围、项目说明等都应保留。没有现成内容的章节，标题保留，正文写 `TODO`。

## 文件归类规则

- 教材、讲义、参考书：放入 `doc/textbook/`。
- 课件、课堂 PPT：放入 `doc/ppt/`。
- 笔记、复习资料、知识点总结：放入 `doc/notes/`。
- 试卷、答案、考纲、期中期末相关材料：放入 `doc/exam/`。
- 实验指导、实验报告、实验模板：放入 `doc/lab/`；实验源码不直接入仓，应以链接形式维护。
- 课程设计、大作业、完整系统、前后端工程：源代码不直接入仓；项目报告、展示文档、说明文档放入 `doc/project/`，外部项目链接写入课程 `README.md` 或 `project/README.md`。
- 单个脚本或小规模代码：只有在它属于课程资料示例、工具脚本或非项目提交源码时，才可放入 `code/`；作业和课程设计源码应改为外部链接。
- 数据文件，如 `.csv`、`.xls`、`.xlsx`、`.json`、`.db`：放入 `data/`，除非它们是某个项目运行所必需的内部文件。
- 图片、图表、截图：放入 `image/`。
- 视频、录屏：放入 `video/`。
- 压缩包：放入 `archive/`，除非压缩包本身是课程提交物或项目交付物的一部分，此时放入对应项目目录。

## 命名规则

- 目录名优先使用英文小写分类名，如 `doc`、`project`、`code`、`data`。
- 课程名、资料文件名可以保留中文，避免无意义重命名。
- 同类材料按课程已有语义命名；如需补充命名，可使用 `年份-类型-说明` 或 `实验编号-说明`。
- 不使用 `新建文件夹`、`资料`、`杂项` 等模糊目录名；确需临时存放时使用 `archive/` 并在 README 中解释。

## 整理流程

1. 进入单门课程目录，列出所有文件和子目录。
2. 读取已有 Markdown、txt 或可读说明文件，确认需要合并到 `README.md` 的内容。
3. 起草或更新课程 `README.md`，先保证说明入口完整。
4. 按文件类型和课程语义创建必要分类目录。
5. 移动资料文件，并同步更新 `README.md` 中的相对路径。
6. 检查是否存在空目录、失效链接、重复说明文件和明显误归类。
7. 使用 `git status` 查看改动范围，确认只改动当前课程相关文件。

## 注意事项

- 不要删除原始课程资料，除非它只是已合并的说明文件或明显重复文件。
- 已入仓但不符合“项目源代码不直接入仓”规范的源码，应迁移为外部链接后从本仓库删除；暂时没有链接时，在 README 中以 `TODO` 标记。
- 不要修改二进制课程资料内容，只做移动和命名整理。
- 对 Git LFS 管理的大文件保持谨慎，移动后需要确认路径变化是否需要同步更新根 README 或 `.gitattributes`。
- 对包含构建产物的项目，优先记录现状；是否清理 `.o`、`.exe`、`node_modules` 等产物应单独决策。
- 每次只整理一门课程，避免大批量移动导致审查困难。
